# Skill: XAML (Robust client UI)

**When to read:** you're writing a client menu/window (BUI menu, lobby panel,
character editor page). For the BUI wiring see `skills/ss14-ui-bui.md`; for
admin EUIs see `skills/ss14-ui-eui.md`.

## Robust XAML basics

Robust uses its own XAML flavor compiled at build time via
`RobustXamlLoader.Load(this)` and `[GenerateTypedNameReferences]` source-gen.
A control is a `partial class` with a paired `.xaml`:

`Content.Client/_Art/Market/UI/MarketSaleMenu.xaml.cs`:

```csharp
using Robust.Client.AutoGenerated;
using Robust.Client.UserInterface.Controls;
using Robust.Client.UserInterface.XAML;

namespace Content.Client._Art.Market.UI;

[GenerateTypedNameReferences]
public sealed partial class MarketSaleMenu : FancyWindow
{
    public Action<string, int>? OnCreateListing;
    public Action<int>? OnCancelListing;

    public MarketSaleMenu()
    {
        RobustXamlLoader.Load(this);
        PriceSpinBox.InitDefaultButtons();
        ListButton.OnPressed += _ => { /* ... */ };
    }

    public void Populate(Dictionary<string, int> contents, List<MarketListingData> listings, int maxListings)
    {
        // rebuild the lists from state
    }
}
```

`MarketSaleMenu.xaml`:

```xml
<controls:FancyWindow xmlns="https://spacestation14.io"
                      xmlns:controls="clr-namespace:Content.Client.UserInterface.Controls"
                      Title="{Loc fork-market-sale-title}"
                      MinSize="400 360">
    <BoxContainer Orientation="Vertical">
        <LineEdit Name="LotNameEdit" />
        <SpinBox Name="PriceSpinBox" />
        <Button Name="ListButton" Text="{Loc fork-market-list-button}" />
        <ItemList Name="ActiveListingsList" />
        <Button Name="CancelButton" Text="{Loc fork-market-cancel-button}" />
    </BoxContainer>
</controls:FancyWindow>
```

## Key conventions

- **Base class**: `FancyWindow` for a titled, styled window (the fork standard).
  `Window` / custom for non-standard. Inherit the existing style.
- **`[GenerateTypedNameReferences]`** — generates typed fields for every
  `Name="..."` in the XAML. In the `.cs`, `ListButton`, `PriceSpinBox`,
  `ActiveListingsList` are real strongly-typed members. **No `FindChild`**.
- **`Name="..."`** on any control you need to touch from code. Use `Access="Public"`
  only if another class needs it (usually not — keep controls private to the
  menu and expose `Action` callbacks instead).
- **Localization**: every `Text`/`Title` the player reads uses `{Loc key}` or
  `{Loc "key" args}`. No literal `Text="Sell"`. See `ss14-localization.md`.
- **`RobustXamlLoader.Load(this)`** as the first line of the constructor.
- **Events**: wire `OnPressed`, `OnItemSelected`, etc. in the constructor;
  expose `Action<...>` fields for the parent (the BUI) to subscribe to. The
  menu doesn't send network messages — it raises `Action`s; the BUI translates
  to `SendMessage`. See `skills/ss14-ui-bui.md`.
- **`Populate(...)` / `UpdateState`-style methods** take the BUI state and
  rebuild the UI. Keep the menu stateless beyond what's needed to render.

## Containers / layout

- `BoxContainer Orientation="Vertical|Horizontal"` — the workhorse layout.
- `GridContainer`, `ScrollContainer`, `SplitContainer` for richer layouts.
- `MarginContainer`, `HSpacer`/`VSpacer` for spacing.
- Set `MinSize="W H"`, `SizeFlagsHorizontal/Vertical` for stretch behavior.

## Common controls

- `Label`, `Button`, `LineEdit`, `TextEdit`, `SpinBox`, `CheckBox`,
  `OptionButton`, `Slider`, `TextureButton`, `TextureRect`, `ItemList`,
  `ProgressBar`. Use `ItemList` for selectable lists (the fork Market uses it
  for listings — see its `OnItemSelected`/`OnItemDeselected` wiring).

## Pitfalls

- **`FindChild("...")` / string lookups** — use `[GenerateTypedNameReferences]`
  + `Name=`.
- **Literal text** — `{Loc ...}` always.
- **Forgetting `RobustXamlLoader.Load(this)`** — the control has no children.
- **Forgetting `[GenerateTypedNameReferences]`** — the typed fields don't exist.
- **Wrong namespace / `xmlns`** — XAML won't compile. Match the fork pattern.
- **Sending messages from the menu** — the menu shouldn't know about
  networking. Raise `Action`s; the BUI sends messages.
- **Rebuilding the whole list on every state push when an incremental update
  would do** — fine for small lists; for large ones, diff. Don't over-engineer
  small UIs though.
